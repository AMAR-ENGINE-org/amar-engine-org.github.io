Overview

AMAR Engine (AME) is a next-generation 3D tool designed to generate virtual worlds from both natural language descriptions and real-world captures. It serves as a decentralized operating system for virtual reality where the physical behavior of objects is governed by a unique Metaclass system.

Core Philosophy

  • Real Mesher, Not Just Models: Unlike NeRF or 3DGS, AME generates true geometric meshes. This ensures that every asset is a "solid" entity compatible with standard 3D environments.
  • Metaclass Logic: Properties are not inherited but composed. A Metaclass defines what an object is and what it can do (e.g., a "Container" Metaclass allows an object to hold liquid).
  • Decentralized Collaboration: AME is a local tool. It uses Cloud-based LLMs to guide the generation process via the Skills Library, overcoming local hardware limitations.
  • Global Identity (AEID): Every scene and asset is assigned a worldwide unique ID (AEID) for registration and tracking in a global database.

Technical Architecture

The Asset vs. The World

  • Assets: Individual entities (like a teacup) with specific Metaclass attributes.
  • Scenes: Entire worlds composed of multiple assets. All scenes are fully compatible and can be combined like meshes in Unreal Engine.

Generation Pipeline

  • Instruction (Skills Library): A RESTful-style specification that allows an AI to guide AME on how to build a scene or asset.
  • Reconstruction: Local AME processes camera footage (photos/videos) or generative prompts to output high-quality Meshes.
  • Attribution: Metaclass schemas are injected into the Mesh to define physical interactions without traditional class inheritance.

Roadmap

  • [ ] Foundation: Define the Metaclass JSON Schema and the Skills RESTful specification.
  • [ ] Local Engine: Develop the core Mesher for image-to-mesh reconstruction.
  • [ ] Global Registry: Launch the AEID database website for asset indexing and sharing.
  • [ ] Ecosystem: Enable seamless scene composition where separate AEIDs function as a unified Metaverse.

Architecture

AMAR Engine is a unified framework designed for multi-path world construction, structural standardization through metaclass inference, and modular physics execution.

High-Level Architecture

Input Layer
↓
Information Pool
↓
AME Admin (Metaclass Inference)
↓
AMAR WORLD (Persistent Standard)
↓
GENIS Runtime Adapter
↓
Genesis Physics Core

Input Layer

The engine supports multiple world construction paths:

  • Scanner-based capture
  • Language-based generation (LLM)
  • Structured model pipelines (e.g., Tahoe)

All inputs generate raw structured information, not finalized world structures.

Information Pool

The Information Pool is a neutral structured domain that contains:

  • Object fragments
  • Spatial hints
  • Semantic signals
  • Raw measurements

It does not contain metaclass assignments, physics bindings, or execution semantics.

AME Admin — Structural Arbitration Layer

AME Admin performs the critical transformation:

Raw Information → Stable Structural Ontology

Responsibilities:

  • Internal model-driven metaclass inference
  • Parameter normalization
  • Structural constraint resolution
  • Dependency validation

AMAR WORLD Standard

AMAR WORLD is the persistent structured output of the system. It is:

  • A directory-based world format
  • Metaclass-resolved
  • Parameter-normalized
  • Dependency-explicit
  • Versionable and portable

Example Structure:

/amar_world
  /objects
  /environment
  /dependencies
  world.meta.json

Runtime Execution Layer

Execution is modular and backend-replaceable:

AMAR WORLD
↓
GENIS Adapter
↓
Genesis Physics Engine
↓
Interactive Simulation

Design Principles

  • Structural permanence over runtime dependency
  • Clear separation of inference and execution
  • Engine-agnostic world specification
  • Modular backend compatibility
  • Extensible metaclass ontology

Core Concepts

1. The Metaclass Protocol

Metaclasses are the genetic genes of the AME system, defining physical properties and behavioral rules for entities.

1.1 Metaclass Evolution Logic

Evolutionary Trend: Concrete → Abstract → Self-consistent

  • Concrete: Extract features and rules from real-world objects
  • Abstract: Abstract specific features into general rules and properties
  • Self-consistent: Form an internally consistent, self-evolving system

1.2 Metaclass Classification

Root Metaclass
  • Definition: Axioms and basic rules of the physical world
  • Examples: Space, time steps, smoothness
  • Characteristics: Non-instantiable, only serving as underlying weights for standard metaclasses
Standard Metaclass
  • Definition: Entity templates with interaction potential
  • Examples: Container, wheel, liquid, solid
  • Characteristics: Possess concrete objects, inherit physical interaction interfaces

1.3 Metaclass Manifestation

Room (Manifestation Space)
  • Definition: Entity existence states in different coordinate systems
  • Examples: Cartesian coordinate system manifestation, relative sensory coordinate system manifestation

2. AEID System (AMAR Engine ID)

AEID system is responsible for generating and managing unique identifiers for entities in the AME system.

2.1 AEID Structure

  • Length: Fixed-length unique identifier
  • Composition: Prefix + timestamp + random number
  • Format: String format, ensuring global uniqueness

3. Skills System

Skills system defines the operations and behaviors that entities can perform.

3.1 Skill Definition

  • Name: Unique identifier of the skill
  • Description: Functional description of the skill
  • Parameters: Parameters required for skill execution
  • Return value: Return value of skill execution
  • Dependencies: Other skills required for skill execution

3.2 Skill Classification

Basic Skills
  • Definition: Basic operations and behaviors
  • Examples: Move, rotate, scale
  • Characteristics: Do not depend on other skills
Composite Skills
  • Definition: Complex operations composed of multiple basic skills
  • Examples: Grab, throw, open
  • Characteristics: Depend on other skills

4. Mesher System

Mesher system is responsible for generating, optimizing, and validating entity meshes.

4.1 Mesh Generation

  • Generate from point clouds
  • Generate from parameters
  • Load from files

5. Admin (Normalization Center)

Admin is the hub of the AME system, responsible for data normalization, metaclass management, and entity instantiation.

6. AMAR GENIS (Physics Engine)

AMAR GENIS is the underlying physics engine of the AME system, responsible for physical simulation and entity behavior.

6.1 Solvers

  • RigidSolver
  • MPMSolver
  • SPHSolver
  • PBDSolver
  • FEMSolver
  • SFSolver
  • ToolSolver

7. AME Scanner

AME Scanner is the 3DGS path executor of the AME system, responsible for spatial skeleton capture.

8. Media Input Path

Media input path is responsible for multimodal attribute sourcing, providing perceptual consistency for entities.

9. LLM API Path

LLM API path is responsible for metaclass definition and logic assignment, providing causality for entities.

10. Export Protocols

10.1 Oasis Domain (osdm)

  • Nature: Folder structure specification
  • Core Purpose: Distributed storage and distribution of interactive worlds

10.2 EME World (emewd)

  • Nature: Binary single file
  • Core Purpose: Specifically for EME project (AI infant evolution project) logic training

System Components

AME Scanner

3DGS path executor responsible for spatial skeleton capture, density collapse processing, and OBB fitting. Provides geometric constraints for entities.

Key Features:

  • Density collapse processing: Strip visual phantoms through non-linear threshold algorithms
  • OBB (Oriented Bounding Box) fitting: Use PCA to decompose the covariance matrix of point clouds
  • Spatial skeleton capture: Extract the spatial skeleton of entities

Media Input Path

Handles multimodal attribute sourcing from cross-spectrum images, temporal videos, and acoustic data. Provides perceptual consistency for entities.

Key Features:

  • Image processing: Analyze images, identify materials, extract colors
  • Video processing: Analyze motion, recognize behavior, perform temporal analysis
  • Audio processing: Recognize acoustic features, analyze environmental audio, calculate acoustic impedance

LLM API Path

Responsible for metaclass definition and logic assignment, converting natural language fragments into Metaclass descriptions and injecting physical constants.

Key Features:

  • Natural language processing: Understand semantics, recognize intent, extract entities
  • Metaclass generation: Generate metaclass definitions from natural language
  • Physical constant injection: Extract and optimize physical parameters

AME Admin

The central hub (Normalization Center) that collapses data from three heterogeneous paths into a single physical entity. Performs metaclass inference and parameter normalization.

Key Responsibilities:

  • Data normalization: Ensure time, space, and semantic consistency
  • Metaclass management: Maintain metaclass registry, perform intelligent classification
  • Entity instantiation: Make final decisions on physical entity instantiation

AMAR WORLD

The persistent structured output of the system, a directory-based world format that is metaclass-resolved, parameter-normalized, and dependency-explicit.

AMAR GENIS

The underlying physics engine that enforces physical laws, implements real-time metaclass injection, and supports topological mutation.

Key Features:

  • Physical handle binding: Implement asynchronous coupling between visual and physical layers
  • Real-time metaclass injection: Enable real-time changes to physical constants
  • Topological mutation support: Real-time collision body reconstruction

Metaclass System

Metaclass Standard

Metaclasses are the genetic genes of the AME system, with the following evolutionary logic:

Evolutionary Trend

Concrete → Abstract → Self-consistent

Root Metaclass

  • Definition: Axioms of the physical world (e.g., space, time steps, smoothness)
  • Characteristics: Non-instantiable, only serving as underlying weights for standard metaclasses

Standard Metaclass

  • Definition: Entity templates with interaction potential (e.g., container, wheel, liquid)
  • Characteristics: Possess concrete objects. When the Scanner sweeps out a volume and the Admin determines it belongs to "Standard Metaclass: Container", the volume immediately inherits the container's physical interaction interface

Manifestation (Room)

  • Entity existence states in different coordinate systems (e.g., Cartesian Room, relative sensory Room)

      "additionalProperties": false

AEID System

AEID Convention

AEID (AMAR Engine ID) is a unique identifier system for entities in the AME ecosystem. It provides a worldwide unique ID for every scene and asset for registration and tracking in a global database.

AEID Structure

  • Length: Fixed-length unique identifier
  • Composition: Prefix + timestamp + random number
  • Format: String format, ensuring global uniqueness

AEID Generation

Example of AEID generation:

class AEIDGenerator {
  generate(): string {
    // 生成前缀
    const prefix = 'AE';
    
    // 生成时间戳
    const timestamp = Date.now().toString(36);
    
    // 生成随机数
    const random = Math.random().toString(36).substring(2, 10);
    
    // 组合生成 AEID
    return `${prefix}_${timestamp}_${random}`;
  }
}

AEID Management

  • Registry: Maintain a registry of all AEIDs
  • Validation: Validate the validity and uniqueness of AEIDs
  • Resolution: Resolve AEIDs to their corresponding entities

AEID Application

  • Entity Identification: Assign unique AEIDs to each entity
  • System Configuration: Associate entity configurations with AEIDs
  • Data Storage: Use AEIDs as keys for data storage
  • Asset Tracking: Track assets across different systems and environments

Skills System

Skills Specification

Skills are the "API of the Virtual World." Instead of rigid code, they provide a standardized way for AI to communicate construction intent to AME.

Key Principles

  • Standardization: Allows AI to understand how to describe virtual world scenes
  • Stateless: Focuses on the "what" and "how" of an object's properties rather than temporal data
  • Extensible: Can be easily extended to support new types of interactions
  • Composable: Multiple skills can be combined to create complex behaviors

Skill Definition

A skill is defined with the following components:

  • Name: Unique identifier of the skill
  • Description: Functional description of the skill
  • Parameters: Parameters required for skill execution
  • Return value: Return value of skill execution
  • Dependencies: Other skills required for skill execution

Skill Classification

Basic Skills
  • Definition: Basic operations and behaviors
  • Examples: Move, rotate, scale
  • Characteristics: Do not depend on other skills
Composite Skills
  • Definition: Complex operations composed of multiple basic skills
  • Examples: Grab, throw, open
  • Characteristics: Depend on other skills

Skill Execution

Example of skill execution:

class SkillExecutor {
  execute(skillName: string, entityId: string, params: any): any {
    // 获取 Skill 定义
    const skill = this.skillRegistry.get(skillName);
    
    // 验证参数
    if (!this.skillValidator.validate(skill, params)) {
      throw new Error('Invalid parameters for skill');
    }
    
    // 执行 Skill
    const result = skill.execute(entityId, params);
    
    // 返回执行结果
    return result;
  }
}

Skill Management

  • Registry: Maintain a registry of all skills
  • Validation: Validate skills and their parameters
  • Composition: Combine multiple skills to form composite skills

API Reference

AMAR Engine provides a command line interface (CLI) tool called `ame` for interacting with the engine.

Installation

# Install dependencies
npm install

# Link the CLI tool (optional)
npm link

Basic Usage

# Initialize AMAR Engine
ame init

# Check engine status
ame status

# Generate a new AEID
ame aeid generate --type asset

# List all registered metaclasses
ame metaclass list

# Register a new skill
ame skill register

Available Commands

System Commands

  • ame init: Initialize AMAR Engine
  • ame status: Check AMAR Engine status

Metaclass Commands

  • ame metaclass register: Register a new metaclass
  • ame metaclass list: List all registered metaclasses
  • ame metaclass compose: Compose multiple metaclasses

AEID Commands

  • ame aeid generate: Generate a new AEID
  • ame aeid validate <aeid>: Validate an AEID
  • ame aeid register <aeid>: Register an AEID
  • ame aeid query <aeid>: Query an AEID

Skill Commands

  • ame skill register: Register a new skill
  • ame skill execute <skillId>: Execute a skill
  • ame skill list: List all registered skills

Examples

Generate and Register an AEID

# Generate a new AEID for an asset
ame aeid generate --type asset

# Validate an AEID
ame aeid validate AST-202602070753-25224137-1EF6

# Register an AEID
ame aeid register AST-202602070753-25224137-1EF6

Register and Execute a Skill

# Register a new skill
ame skill register

# Execute a skill
ame skill execute create_asset

# List all skills
ame skill list

Manage Metaclasses

# Register a new metaclass
ame metaclass register

# List all metaclasses
ame metaclass list

# Compose multiple metaclasses
ame metaclass compose
